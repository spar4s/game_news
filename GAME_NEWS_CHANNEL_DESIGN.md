# 游戏资讯频道系统设计文档

## 1. 项目目标

构建一个类似 `Horizon` 的游戏资讯聚合与分发系统，用于为“每日游戏资讯频道”自动收集、筛选、总结和发布内容。

系统目标：

- 自动采集多来源游戏资讯
- 对资讯进行去重、聚类和优先级排序
- 生成适合频道发布的中文摘要
- 支持每日固定时段产出“游戏日报”
- 支持将高优先级新闻实时推送到频道
- 为后续扩展到网站、邮件、Telegram、Discord、公众号预留接口

## 2. 产品定位

该系统不是通用 RSS 阅读器，而是一个“游戏垂类资讯编辑助手”。

与普通聚合器的区别：

- 聚焦游戏行业，不追求全网资讯覆盖
- 优先保留对频道观众有价值的内容，而不是所有更新
- 强调“能不能发”“值不值得发”“应该怎么发”
- 输出的结果不是原始链接列表，而是可直接发布的成品内容

## 3. 核心使用场景

### 3.1 每日频道日报

每天固定时间自动生成一份中文游戏资讯日报，适合发布到频道。

示例结构：

1. 今日重点 3 条
2. 重要行业动态 3-5 条
3. 游戏更新与发售资讯 5-10 条
4. 传闻与社区热点 2-3 条

### 3.2 快讯模式

当抓到高优先级资讯时，系统立即生成短摘要并推送到频道。

适用场景：

- 新作正式公开
- 发售日公布
- 大型更新上线
- 停服、裁员、工作室关闭
- 大厂财报和战略调整

### 3.3 编辑辅助模式

运营者每天查看系统筛好的候选资讯，手动选择需要发布的条目，再一键生成最终稿件。

## 4. 目标用户

- 个人游戏资讯频道运营者
- 游戏媒体编辑或内容作者
- Discord / Telegram 游戏社区管理员
- 需要跟踪游戏行业动态的从业者

## 5. MVP 范围

第一阶段只做最小可用版本，优先跑通主流程。

### 5.1 MVP 功能

- 支持固定游戏资讯源抓取
- 支持 RSS / Atom 抓取
- 支持简单网页抓取
- 支持 SQLite 存储
- 支持文章去重
- 支持 AI 打分和中文摘要
- 支持日报生成
- 支持导出 Markdown
- 支持 Telegram 或 Discord 二选一发布

### 5.2 MVP 不做

- 多用户系统
- 复杂后台管理界面
- 浏览器插件
- App
- 高级个性化推荐
- 自动视频生成
- 社交平台复杂互动分析

## 6. 资讯源设计

资讯源分为四类：官方源、媒体源、社区源、平台源。

### 6.1 官方源

优先级最高，可信度最高。

建议首批接入：

- PlayStation Blog
- Xbox Wire
- Nintendo News
- Steam News
- Epic Games News
- Riot Games News
- Blizzard News
- HoYoverse / 米哈游公告
- 腾讯游戏官方公告
- 网易游戏官方公告

### 6.2 媒体源

用于补充试玩、采访、爆料和行业信息。

建议首批接入：

- IGN
- GameSpot
- Gematsu
- Video Games Chronicle
- Eurogamer
- Rock Paper Shotgun
- 机核
- 游民星空
- 3DM
- 篝火营地

### 6.3 社区源

用于发现热点讨论和高关注更新。

建议接入：

- Reddit 指定板块
- Steam 社区公告
- ResetEra 热门帖
- Bilibili 游戏区动态
- YouTube 官方频道视频更新

### 6.4 平台源

用于监控上架、促销、补丁和新品。

建议接入：

- Steam 新品与新闻
- App Store / Google Play 重点游戏更新
- Epic Store 新闻页

## 7. 系统流程

整体流程如下：

1. 定时抓取资讯源
2. 标准化为统一 Article 结构
3. 基于标题、URL、正文摘要做去重
4. 对同一事件做聚类
5. 调用 AI 进行分类、打分、摘要和背景补充
6. 根据规则生成“日报候选池”
7. 输出为 Markdown / HTML / 频道消息
8. 调用发布器推送到指定渠道

## 8. 信息处理逻辑

### 8.1 标准化

每条资讯统一转为以下字段：

- `id`
- `source_name`
- `source_type`
- `title`
- `url`
- `author`
- `published_at`
- `fetched_at`
- `language`
- `raw_content`
- `summary`
- `tags`
- `platforms`
- `games`

### 8.2 去重

去重分三层：

- URL 去重：相同链接直接视为重复
- 标题相似度去重：相似标题合并
- 语义去重：AI 判断是否为同一事件

### 8.3 聚类

把同一事件的多条资讯归并为一个 Topic。

例如：

- 官方预告片发布
- IGN 试玩前瞻
- Gematsu 新闻稿
- Reddit 讨论串

最终聚成一个“某游戏新情报公开”的 Topic。

### 8.4 分类

建议分类标签：

- 新作公布
- 发售日期
- 更新补丁
- DLC / 扩展内容
- 财报与销量
- 工作室动态
- 停服与维护
- 联动活动
- 传闻爆料
- 社区热点

### 8.5 评分

每条 Topic 输出 0-100 分的频道价值分。

评分维度：

- 来源可信度
- 新闻重要性
- 时效性
- 用户关注度
- 与频道定位匹配度
- 是否重复报道

建议权重：

- 可信度：25
- 重要性：25
- 时效性：20
- 匹配度：20
- 讨论热度：10

## 9. AI 能力设计

AI 不直接替代规则，而是作为规则增强器。

### 9.1 AI 任务

- 标题翻译
- 中文摘要
- 游戏名识别
- 平台识别
- 事件分类
- 重要性评分
- 背景知识补充
- 多源聚类辅助判断

### 9.2 输出格式

建议强制 AI 返回 JSON，避免解析不稳定。

示例字段：

- `headline_zh`
- `brief_summary`
- `importance_score`
- `category`
- `confidence`
- `game_titles`
- `platforms`
- `why_it_matters`
- `source_reliability`

### 9.3 AI 使用原则

- 先规则后 AI，降低成本
- 对高优先级内容再调用更昂贵模型
- 关键字段强约束输出格式
- 保留原始数据，便于回溯和重新生成

## 10. 数据模型

### 10.1 `sources`

- `type`
- `url`
- `enabled`
- `priority`
- `parser_type`
- `language`

### 10.2 `articles`

字段建议：

- `id`
- `source_id`
- `title`
- `url`
- `content`
- `published_at`
- `fetched_at`
- `hash`

### 10.3 `topics`

字段建议：

- `id`
- `title`
- `category`
- `importance_score`
- `summary`
- `why_it_matters`
- `published_at`

### 10.4 `topic_articles`

## 11. 技术架构

建议使用 Python 实现，因为生态成熟，和 Horizon 风格也接近。

### 11.1 技术栈

- 语言：Python 3.12+
- 抓取：`httpx`, `feedparser`, `beautifulsoup4`
- 数据库：SQLite + `sqlalchemy`
- 任务调度：`APScheduler` 或 GitHub Actions / cron
- 配置管理：`pydantic-settings`
- AI 调用：OpenAI 兼容 SDK
- 模板生成：`jinja2`
- 渠道发布：Telegram Bot API / Discord Webhook
- 日志：`loguru` 或 `structlog`

### 11.2 模块划分

建议目录：

```text
game-news-bot/
  app/
    collectors/
    parsers/
    pipelines/
    scoring/
    ai/
    delivery/
    models/
    storage/
    templates/
    config/
    cli/
  data/
  tests/
  docs/
```

### 11.3 模块职责

- `collectors/`：抓取 RSS、网页、社区内容
- `parsers/`：清洗正文和提取元数据
- `pipelines/`：去重、聚类、排序
- `scoring/`：规则打分与 AI 打分融合
- `ai/`：统一封装模型调用
- `delivery/`：推送到频道
- `storage/`：数据库访问
- `cli/`：命令行入口

## 14. 配置设计

建议采用 `yaml` 或 `json` 配置。

核心配置项：

- AI 模型与 API Key
- 资讯源清单
- 频道推送配置
- 打分阈值
- 日报时间
- 快讯阈值
- 语言偏好
- 频道定位偏好

频道定位偏好示例：

- 偏主机新闻
- 偏 PC / Steam
- 偏二次元手游
- 偏国产单机
- 偏行业资本与公司动态

## 15. 频道策略设计

这个系统最关键的是“筛选逻辑要贴合你的频道”。

建议把频道策略做成独立配置层。

### 15.1 频道画像字段

- 主要受众
- 偏好平台
- 偏好游戏类型
- 是否接受传闻
- 是否强调行业新闻
- 是否强调手游活动
- 每日发文数量上限

### 15.2 示例策略

如果你的频道是“综合游戏日报”：

- 保留官宣、发售、更新、行业动态
- 降低纯社区争议内容权重
- 允许保留高热度传闻，但必须标记为传闻

如果你的频道是“Steam / PC 向”：

- Steam 新品、补丁、Mod、EA 更新加权
- 主机独占新闻降权

- 增加快讯推送

### 阶段 3：产品化

- 增加简单 Web 面板
- 支持多频道策略
- 支持历史检索与归档

## 17. 风险与难点

### 17.1 数据源稳定性

很多站点结构会变，网页解析容易失效。

应对方式：

- 优先 RSS
- 把 parser 独立出来
- 记录失败日志和重试

### 17.2 AI 输出不稳定

应对方式：

- 强制 JSON 输出
- 增加 schema 校验
- 失败时降级为规则摘要

### 17.3 重复新闻很多

游戏媒体经常转译同一条官宣。

应对方式：

- 优先官方源
- 媒体源作为补充材料
- 引入事件聚类机制

### 17.4 频道风格不统一

应对方式：

- 提前定义模板
- 把频道画像做成配置
- 所有摘要都走统一格式器

## 18. 推荐实施路线

如果目标是尽快上线，建议按这个顺序做：

1. 建立项目骨架

5. 做 AI 摘要

7. 接 Telegram 或 Discord 发布

## 20. 下一步建议

从落地角度，下一步最合理的是直接开始做 MVP 工程骨架。

建议优先实现这 4 个命令：

- `fetch`：抓资讯
- `process`：去重和打分
- `digest`：生成日报
- `publish`：推送到频道

这样我们可以先把整条链路跑通，再逐步细化。
